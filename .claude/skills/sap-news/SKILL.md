# sap-news — мониторинг перехода компаний РФ с SAP

## Когда использовать
Пользователь говорит: «новости по SAP», «что нового по переходу с SAP», «сделай дайджест», «проверь новости за неделю», «что по такой-то компании», «/sap-news». Всё, что связано с мониторингом темы импортозамещения SAP в России.

## Режимы работы

Скил понимает аргументы после `/sap-news`:

- `/sap-news` — дайджест за последние 7 дней от сегодняшней даты.
- `/sap-news --from YYYY-MM-DD` — с указанной даты по сегодня.
- `/sap-news --from YYYY-MM-DD --to YYYY-MM-DD` — произвольный период.
- `/sap-news --company <имя>` — всё по конкретной компании (например `--company Росатом`). Период — с начала текущего года, если не указан иначе.
- `/sap-news --sector <название>` — по отрасли: `нефтегаз`, `ритейл`, `финсектор`, `металлургия`, `атом`, `энергетика`, `телеком`, `промышленность`.
- `/sap-news --help` — вывести справку по режимам.

Если аргументы противоречат друг другу или непонятны — спросить уточнение, не запускать поиск вслепую.

## Алгоритм выполнения

### Шаг 1. Определить период
Если период не задан — последние 7 дней от `today`. Записать диапазон в переменные `from` и `to`.

### Шаг 2. Собрать запросы
Использовать шаблоны из `queries.md`. Для каждого режима:
- **Default / --from / --to**: 6–8 широких запросов по базовым формулировкам.
- **--company X**: 3–4 запроса с именем компании в разных формах.
- **--sector X**: 4–5 запросов с ключевыми словами отрасли.

Запросы запускать ПАРАЛЛЕЛЬНО одним сообщением (несколько вызовов WebSearch в одном response).

### Шаг 3. Фильтрация по источникам
После получения результатов каждого запроса — отфильтровать URL:
- **Оставить** домены из раздела A/B/C в `sources.md` (ядро, регуляторика, отраслевые).
- **Пропустить** домены из раздела E (шум: продающие страницы, топы аналогов от интеграторов).
- **Блог интегратора** (раздел D в sources.md) — оставить ТОЛЬКО если в заголовке есть имя конкретной компании-клиента (это кейс, не маркетинг).

### Шаг 4. Дедупликация
1. Прочитать `news-archive/seen-urls.txt` (если не существует — считать пустым).
2. Отбросить URL, которые уже есть в архиве.
3. В рамках текущей выдачи — отбросить дубли по URL и по нормализованному заголовку (первые 60 символов в нижнем регистре без знаков препинания).

### Шаг 5. Проверить даты
Отсечь новости вне периода `from..to`. Дата публикации часто видна в URL (например `/2026-02-06_...`) или в тексте результата. Если дата не определяется — оставить материал, но пометить `[дата неизвестна]` в дайджесте.

### Шаг 6. Классификация
Каждую новость отнести к одной категории:
- **Кейсы компаний** (кто переходит, куда, сроки, бюджет) — главная ценность.
- **Регуляторика** (законы, постановления, КИИ, штрафы, дедлайны).
- **Рынок и аналитика** (исследования, прогнозы, размер рынка, рейтинги).
- **Мнения и критика** (комментарии экспертов, интервью, скептические материалы).
- **Продукты и вендоры** (новые релизы, консорциумы, M&A среди российских разработчиков).

### Шаг 7. Сформировать дайджест
Использовать шаблон из раздела «Формат дайджеста» ниже. Сохранить в `news-archive/digest-YYYY-MM-DD.md` (имя по дате запуска). Если файл с таким именем уже есть — дописать суффикс `-2`, `-3` и т.д.

### Шаг 8. Обновить архив URL
Добавить URL всех включённых в дайджест новостей в `news-archive/seen-urls.txt` (по одному URL на строку). Не сортировать, просто дописывать в конец — так видна хронология.

### Шаг 9. Вывести краткий отчёт в чат
Формат:
```
Дайджест сохранён: news-archive/digest-YYYY-MM-DD.md
Период: YYYY-MM-DD .. YYYY-MM-DD
Найдено: N новостей (M новых после дедупликации)

Топ-5:
1. [заголовок] — [источник]
2. ...

Категории: кейсы — N, регуляторика — N, рынок — N, мнения — N, продукты — N.
```

## Формат дайджеста (шаблон)

```markdown
# Дайджест SAP-news: YYYY-MM-DD .. YYYY-MM-DD

Сформирован: YYYY-MM-DD HH:MM
Всего новостей: N (из них новых: M)

## Кейсы компаний
### [Компания] — [короткая суть]
- **Источник:** [название](url)
- **Дата:** YYYY-MM-DD
- **Суть:** 2–3 предложения. Что происходит, какой вендор заменяется, на что, сроки, бюджет если есть.
- **Теги:** #отрасль #вендор-замена

## Регуляторика
### [Заголовок события]
- **Источник:** [название](url)
- **Дата:** YYYY-MM-DD
- **Суть:** ...

## Рынок и аналитика
...

## Мнения и критика
...

## Продукты и вендоры
...

## Источники без даты / к доработке
Список материалов, где не удалось определить дату или достоверность.
```

Внутри категории — сортировка по дате публикации (свежие сверху).

## Правила, которые нельзя нарушать

1. **Всегда проверять архив `seen-urls.txt` перед формированием дайджеста** — чтобы не повторять новости из прошлых прогонов.
2. **Не домысливать содержание новости** — если в результатах WebSearch мало контекста, сделать WebFetch на URL и вытащить суть. Не писать «предположительно» в дайджесте.
3. **Отсекать рекламные продающие страницы** — даже если они попали в выдачу по релевантности. Список таких доменов — в `sources.md` раздел E.
4. **Не путать SAP как продукт и SAP как компанию** — запрос «SAP закрыл офис» ≠ «компания перешла с SAP». В дайджест попадают именно кейсы перехода/замещения, а не корпоративные новости SAP.
5. **Уважать периодичность** — если пользователь просит за неделю, а в архиве последний дайджест вчерашний, тактично подсветить: «за 1 день найдено N новостей, возможно имеет смысл расширить период».
6. **При создании скрипта или автоматизации — НЕ использовать.** Скил работает через WebSearch + ручную фильтрацию мной (Claude). Никаких самописных RSS-парсеров и cron-скриптов в рамках этого скила.

## Файлы скила
- `SKILL.md` — этот файл.
- `sources.md` — список источников с классификацией (A/B/C/D/E).
- `queries.md` — шаблоны поисковых запросов для разных режимов.

## Файлы архива (в `news-archive/` в корне проекта)
- `seen-urls.txt` — список уже показанных URL (для дедупликации).
- `digest-YYYY-MM-DD.md` — сохранённые дайджесты.

## Связанные действия
После первого ручного прогона скила имеет смысл поставить его на расписание через `/schedule` — например, каждый понедельник утром запускать `/sap-news` за прошедшую неделю. Об этом напомнить пользователю после успешного первого запуска.
